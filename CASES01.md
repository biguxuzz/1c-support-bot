# Анализ: кейсы для автоматической обработки и требуемые данные

Источник: `.wikis/ikon_wiki_support/wiki/` — 236 обращений из Jira Service Desk (ERPWE-561 — ERPWE-6207, фев 2025 — март 2026).

---

## 1. Управление доступом (~45%) — наивысший потенциал автоматизации

### А. Открытие/закрытие периодов (регулярная, шаблонная операция)

| Что нужно из базы | Зачем |
|---|---|
| Список пользователей с ролями и группами доступа | Определить, кому закрыть/открыть доступ |
| Признак «бухгалтер» в ролях пользователя | Исключить бухгалтеров из массового закрытия |
| Текущее состояние периодов (какие открыты/закрыты) | Не закрыть уже закрытый или не открыть уже открытый |
| Список исключений (кому оставить доступ) | Из запроса пользователя + исторический список |

### Б. Создание учётных записей новым сотрудникам

| Что нужно из базы | Зачем |
|---|---|
| Список существующих пользователей с ФИО | Найти «аналогичного» сотрудника по запросу «по аналогии с ...» |
| Полный набор ролей/групп «образцового» пользователя | Скопировать права на нового |
| Структура организаций и подразделений | Привязать нового к нужной организации |
| Признак «уже существует» по ФИО | Предотвратить дублирование |

### В. Сброс паролей

| Что нужно из базы | Зачем |
|---|---|
| Существует ли пользователь с таким ФИО | Не сбрасывать пароль несуществующему |
| Текущий статус учётной записи (заблокирована/активна) | Диагностировать причину |

### Г. Выдача ролей/групп доступа

| Что нужно из базы | Зачем |
|---|---|
| Текущие роли и группы пользователя | Не добавлять то, что уже есть |
| Полный справочник ролей с описанием | Распознать «мне нужен доступ к складу» → «Специалист по учёту складских операций» |
| Маппинг: роль → какие объекты/действия открывает | Для проверки и объяснения пользователю |
| Группы «НСИ для RLS» по категориям номенклатуры | Определить нужную группу по контексту запроса |

## 2. Интеграция ERP ↔ КУФИБ (~20%)

### А. Проверка статуса интеграции документа

| Что нужно из базы | Зачем |
|---|---|
| Статус обмена (последний запуск, результат) | Ответить «интеграция идёт, ожидайте до 24 мин» |
| Очередь невыгруженных документов | Оценить объём и предполагаемое время |
| Лог ошибок обмена (последние ошибки) | Диагностировать причину |

### Б. Дублирование документов при миграции

| Что нужно из базы | Зачем |
|---|---|
| Настройки интеграции (правила обмена, расписания) | Проверить, не «слетели» ли настройки |
| Список документов в ERP с признаком интеграции | Найти дубли по номеру/дате/сумме |
| Соответствие ERP ↔ КУФИБ по GUID/номеру | Сопоставить пары |

### В. Документ без проводок после миграции

| Что нужно из базы | Зачем |
|---|---|
| Документ: статус проведения, наличие проводок | Определить, нужно ли ручное проведение |
| Причина непроведения (сообщение об ошибке) | Отрицательные остатки? Неверные реквизиты? |

---

## 3. Ошибки проведения документов (~15%)

### А. Отрицательные остатки

| Что нужно из базы | Зачем |
|---|---|
| **Оперативные остатки** по складу/номенклатуре (текущий момент, НЕ на дату документа) | Определить, чего и сколько не хватает |
| Документ, который не проводится: тип, дата, склад, номенклатура, количество | Сопоставить потребность с остатками |
| Очередь непроведённых документов в хронологии | Понять, может ли перепроведение ранее датированного документа решить проблему |
| Перемещения «в пути» на этот склад | Ожидаемые поступления |

### Б. Сборка/разборка: не подтягивается материал

| Что нужно из базы | Зачем |
|---|---|
| Спецификация (BOM): состав, действующая версия | Проверить, привязана ли спецификация |
| Флаг «Документ содержит ручные корректировки» | Если включён — списание не контролируется |
| Остатки компонентов для сборки | Хватает ли комплектующих |

## 4. Инвентаризация (~5%)

### А. Расхождения при инвентаризации

| Что нужно из базы | Зачем |
|---|---|
| Данные документа инвентаризации: учётное количество, фактическое | Определить расхождение |
| Хронология документов (даты пересчёта vs перемещения vs пересортицы) | ERPWE-4109: разница дат = причина расхождения |
| Остатки по конкретным складам/позициям | Верифицировать данные |

---

## 5. Закрытие периода и себестоимость (~5%)

### А. Проблемы при закрытии месяца

| Что нужно из базы | Зачем |
|---|---|
| Статус закрытия месяца по периодам | Предварительное vs окончательное |
| Результаты расчёта себестоимости (ошибки, незакрывшиеся счета) | Диагностировать причину |
| Счета с ненулевыми остатками после закрытия | ERPWE-5408: не закрывается 25 счёт |
| Регламентные задания: расписание, последний запуск, результат | Проверить, выполнялось ли предварительное закрытие |

### Б. ГФУ (группы финансового учёта)

| Что нужно из базы | Зачем |
|---|---|
| Текущие назначения ГФУ по номенклатуре | Что изменено |
| Количество документов, затрагиваемых изменением ГФУ | Оценить объём перепроведения |
| Период, с которого вступают изменения | Не менять задним числом |

---

## 6. НСИ и номенклатура (~5%)

### А. Проверка перед изменением единицы измерения

| Что нужно из базы | Зачем |
|---|---|
| Количество документов, использующих данную номенклатуру | Если > 0 — **запретить** изменение, предложить создать новую |
| Текущая единица измерения номенклатуры | Предупредить о риске |

### Б. Создание новых позиций номенклатуры

| Что нужно из базы | Зачем |
|---|---|
| Структура справочника (виды, категории, атрибуты, бренды, ТНВЭД) | Предложить правильные значения |
| Группы «НСИ для RLS» пользователя | Проверить, есть ли право создавать в данной категории |
| Существующие аналогичные позиции | Не создать дубль |

## 7. Отчёты и сверки (~3%)

| Что нужно из базы | Зачем |
|---|---|
| Права пользователя на конкретный отчёт | Ответить «нет доступа» или «отчёт доступен по пути ...» |
| Маппинг: отчёт → группа доступа | Автоматически добавить в нужную группу |

---

## 8. Технические проблемы (~2%)

| Что нужно из базы | Зачем |
|---|---|
| Статус серверных служб 1С | Ответить, работает ли сервер |
| Свободное место на дисках сервера | ERPWE-3199: нехватка места |
| Логи последних сбоев | Диагностика |
| Текущая версия конфигурации | Проверить необходимость обновления |

## Итоговая сводка по автоматизируемости

| Категория | Доля | Автоматизируемость | Ключевые данные |
|---|---|---|---|
| Доступ: периоды | ~20% | **Высокая** | Пользователи, роли, состояние периодов |
| Доступ: новые сотрудники | ~10% | **Высокая** | ФИО, роли «образца», организации |
| Доступ: пароли | ~5% | **Высокая** | Наличие пользователя |
| Доступ: роли/отчёты | ~10% | **Средняя** | Справочник ролей, текущие права |
| Интеграция | ~20% | **Средняя** | Лог обмена, статусы документов |
| Проведение/остатки | ~15% | **Средняя** | Оперативные остатки, спецификации |
| Закрытие периода | ~5% | **Низкая** | Результаты закрытия, себестоимость |
| НСИ | ~5% | **Средняя** | Использование номенклатуры в документах |
| Инвентаризация | ~5% | **Низкая** | Хронология документов, остатки |
| Отчёты | ~3% | **Высокая** | Маппинг отчёт→группа |
| Технические | ~2% | **Низкая** | Статус серверов, логи |

**~45% обращений** (доступ, пароли, отчёты) — высокий потенциал полной автоматизации при наличии доступа к справочникам пользователей, ролей и групп доступа.

**~35%** (интеграция, проведение, НСИ) — частичная автоматизация (диагностика + рекомендация), требующая чтения оперативных остатков, логов обмена и статусов документов.

**~20%** (закрытие периода, инвентаризация, технические) — преимущественно консультационные, требуют сложной аналитики и экспертного решения.
